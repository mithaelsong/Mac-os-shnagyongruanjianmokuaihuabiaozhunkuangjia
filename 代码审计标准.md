# 代码审计标准（仙人指路·商用模块标准框架）
> 2026-06-05 08:30 晓筱发布

---

## 一、功能完整性审计

每条功能逐条检查：
1. **功能清单核对** — 设计文档要求的功能点是否全部实现，没有遗漏
2. **接口签名检查** — 方法名、参数、返回值是否与设计一致
3. **边界处理** — 空值、重复注册、不存在、非法输入时是否正确处理
4. **错误处理** — 失败时是否返回合理的错误值/状态，不崩溃

## 二、代码质量审计

1. **线程安全** — `os_unfair_lock` / `NSLock` 使用是否正确
   - ❌ 锁内调用外部回调 / 发通知
   - ❌ 锁嵌套导致死锁
   - ✅ 锁范围合理，只保护共享数据
2. **合理性（绕弯子检查）** — 功能实现是否有更简洁的方法
   - 比如功能22字体加载走 CoreText → NSFont 两步，其实可以一次搞定
   - 比如功能24证书检查手动解析 DER hex，其实有现成 API
3. **命名规范** — 类名、方法名、变量名是否统一、可读
4. **注释** — 公开方法是否有注释，注释是否与实现一致
5. **测试覆盖** — 测试是否覆盖正常/边界/异常/并发

## 三、架构一致性审计

1. **设计模式统一** — 全部用单例还是允许工厂/构造器注入？
2. **API风格统一** — 注册方法名是 `registerXX` 还是 `setXX`？是否统一？
3. **跨模块依赖** — 功能之间的引用关系（如A调用B）是否有循环依赖？
4. **对外接口统一** — 所有模块暴露的协议风格是否一致

## 四、安全性审计

1. **签名验证** — 功能24的证书检查逻辑是否可靠
2. **崩溃隔离** — 功能26的 `execute` 异常捕获是否覆盖所有场景
3. **沙盒支持** — 功能25的沙盒检测是否准确

## 五、语言规则

整个框架中所有可见文字全部用中文：

| 类别 | 规则 |
|------|------|
| 测试输出（print + fatalError） | → 中文 |
| 日志消息（logger.info/warning/error） | → 中文 |
| 注释（`///` doc comment、`//` 普通注释、`// MARK:`标注） | → 中文 |
| 使用示例（文件末尾 `/* ... */` 演示注释块） | → 中文 |
| 代码标识符（类名、方法名、变量名、协议名、枚举case等） | → 保持英文（Swift语言规则，改成中文无法编译） |

## 审计执行流程

1. 逐文件读取全部代码
2. 按五大项逐行检查
3. 发现问题先记录、再修复
4. 修复完重新审计一遍
5. 零问题后标记为"审计通过"
